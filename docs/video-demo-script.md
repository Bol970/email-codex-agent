# Video Demo Script

Готовый сценарий для записи учебного проекта Email Codex Agent.

Цель видео: за 5-7 минут показать, что это локальный веб-сервис для работы с AgentMail-почтой, где пользователь читает письма, а встроенный Codex помогает суммировать, переводить, классифицировать и готовить черновики без автоматической отправки.

## Перед записью

1. Открыть проект на отдельном мониторе во весь экран:
   `http://127.0.0.1:5173`
2. Убедиться, что в `/api/status` включён privacy-mode:
   `blurEmailAddresses: true`
3. Убедиться, что в интерфейсе не видно реальных email-адресов. Они должны быть размыты.
4. Масштаб браузера: 100% или 110%.
5. Закрыть лишние вкладки, уведомления и терминалы с секретами.

## Роли во время записи

Пользователь:

- открывает браузер во весь экран;
- включает запись экрана;
- больше ничего не делает, кроме случаев, когда нужно подтвердить действие мышью.

Codex:

- ведёт презентацию по сценарию;
- говорит, какой элемент сейчас показываем;
- запускает безопасные действия: перевод, summary, draft-first workflow;
- не отправляет письма автоматически.

> Это учебный локальный сервис Email Codex Agent. Он работает как веб-приложение на localhost и подключается к одному AgentMail inbox. Слева находится список фильтров и inbox, в центре список писем, справа область чтения и панель встроенного Codex-агента.

Показать:

- список писем;
- область чтения;

### 0:25 - 1:05. Локальность и приватность

Что на экране: sidebar и любое письмо с размытыми адресами.

Текст:

> Важная часть проекта - он рассчитан на локальную работу. Сервис поднимается на 127.0.0.1, а email-адреса можно блюрить через env-параметр BLUR_EMAIL_ADDRESSES. Это удобно для демонстраций и записи видео: логика работает с настоящими данными, но на экране приватные адреса не читаются.

Показать:

- адрес inbox в sidebar размытый;
- from/to в письме размытые;
- текст письма остаётся читаемым.

### 1:05 - 1:45. Чтение входящего письма

Что сделать:

1. Открыть одно входящее письмо.
2. Показать заголовок, количество сообщений, текст письма.
3. Показать, что unread-индикатор пропадает после открытия.

Текст:

> Здесь обычный mail-client workflow: я выбираю письмо, читаю содержимое, вижу метки и могу вручную ответить. При открытии письма оно сразу помечается как прочитанное, поэтому индикатор unread исчезает без дополнительного клика.

### 1:45 - 2:35. Перевод письма на русский

Что сделать:

1. Нажать кнопку перевода на русский в панели Codex.
2. Дождаться ответа в диалоге.

Текст:

> В Codex-панели есть preset-действия. Например, можно перевести выбранный thread на русский. Важно, что письмо передаётся агенту как недоверенный контекст: он должен использовать его как данные, а не выполнять инструкции из самого письма.

Если ответ долго идёт:

> Ответ стримится через Codex app-server. Если модель думает дольше, это нормально: локальный UI остаётся доступным, а результат появится прямо в диалоге.

### 2:35 - 3:25. Суммаризация и важные пункты

Что сделать:

1. Нажать Summarize.
2. Показать краткое резюме в Codex-чате.

Текст:

> Второй сценарий - быстро понять смысл входящего письма. Codex выделяет, кто написал, зачем написал, есть ли срочность, риски и что лучше сделать дальше. Это превращает длинное письмо в короткий triage-блок.

### 3:25 - 4:25. Черновик ответа, draft-first policy

Что сделать:

1. Нажать Draft reply.
2. Дождаться создания черновика.
3. Показать, что черновик появился в письме.
4. Не нажимать Send.

Текст:

> Главное ограничение безопасности в первой версии - draft-first. Codex может подготовить черновик, но не может сам отправить письмо. Отправка остаётся только через явный пользовательский клик. Это делает агента полезным, но не даёт ему автономно совершать необратимые действия.

Показать:

- сообщение "Черновик создан в выбранном письме";
- карточку draft под письмом;
- кнопки Send и Delete draft.

### 4:25 - 5:15. Подтверждения как в Codex

Что сделать:

1. Если появляется запрос подтверждения, показать его прямо в диалоге.
2. Нажать "Подтвердить" или "Отклонить" в зависимости от действия.

Текст:

> Подтверждения встроены прямо в чат, как в Codex. Если агент хочет выполнить чувствительное действие, он не уводит пользователя в отдельную техническую вкладку, а показывает вопрос в диалоге. После решения видно, что именно было подтверждено или отклонено.

Если approval не появился:

> В этом сценарии approval может не появиться, потому что mail-tools ограничены безопасным набором. Но интерфейс уже поддерживает inline approval flow.

### 5:15 - 6:00. Ручной ответ

Что сделать:

1. Ввести короткий текст в Write a reply.
2. Нажать Draft, а не Send.
3. Показать новый черновик.

Текст:

> Помимо агента, пользователь может работать вручную. Можно написать ответ самому и сохранить его как черновик. Это важно: приложение не заменяет mail-client, а добавляет агентный слой поверх привычного workflow.

### 6:00 - 6:40. Архитектура

Что на экране: можно оставить общий вид приложения.

Текст:

> Архитектурно это один локальный Node-сервис. Express отдаёт API и фронтенд, AgentMail используется для inbox, threads, messages, drafts и labels. Для realtime-событий используются локальные SSE/WebSocket-потоки, а Codex запускается через codex app-server. ChatGPT-подписка используется через локальную авторизацию Codex, без отдельного OpenAI API key.

### 6:40 - 7:00. Финал

Текст:

> В итоге получился локальный email workspace: пользователь читает и отвечает на письма сам, а Codex помогает с переводом, summary, triage и черновиками. Все опасные действия остаются под явным контролем пользователя.

1. 0:00 - 0:20: что это за проект.
2. 0:20 - 0:45: локальность и blur email.
3. 0:45 - 1:20: открыть письмо и показать чтение.
4. 1:20 - 2:00: Translate to Russian или Summarize.
5. 2:00 - 2:40: Draft reply и draft-first policy.
6. 2:40 - 3:00: финал про безопасность и архитектуру.

## Команды для проверки перед записью

```bash
npm run lint
npm run test
npm run build
curl -sS http://127.0.0.1:5173/api/status
```

Ожидаемый status должен содержать:

```json
{
  "mode": "live",
  "blurEmailAddresses": true,
  "codexAvailable": true
}
```

## Что лучше не показывать

- `.env.local`
- терминалы с ключами или историей команд;
- реальные email-адреса без blur;
- отправку письма через Send, если это не специально подготовленный test inbox.

## План на запись с Codex-оператором

1. Пользователь открывает проект на отдельном мониторе и включает запись.
2. Пользователь пишет в чат: `начали`.
3. Codex пошагово даёт короткие команды: что показать, что нажать, что подождать.
4. Если нужен автономный режим без ручных кликов, следующий технический шаг - добавить в проект demo-runner или browser automation script, который сам кликает по UI по этому сценарию.
